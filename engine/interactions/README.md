# Structured interactions

The interaction package is a platform-neutral capability for authored games. It turns an authored interaction definition into a deterministic selection window and a validated resolution. Surfaces can render the menus and events on phones, shared screens, voice clients, or other interfaces without the engine knowing which surface is in use.

## Vocabulary

- InteractionDefinition: authored option catalog and player-count allowance rules.
- InteractionWindow: one active round for one interaction definition.
- InteractionOption: one authored player-facing choice, optionally gated by evidence.
- InteractionTarget: an eligible character, object, location, or other game entity.
- InteractionSelection: a player's locked target and option.
- InteractionResolution: deterministic ordered selections plus public groups and private selections.
- Private feedback: an event addressed only to the player who made a private selection.

## Lifecycle

1. The engine opens a window with the round, seating order, participants, eligible targets, and each player's held evidence.
2. The engine builds each menu from authored baseline options and evidence requirements. Menus are deterministic and never generated by a model.
3. Players submit one or more selections until their count-specific allowance is exhausted. A selection can be revised before lock by supplying its selection ID.
4. The engine locks the window, rotates resolution order from the round index, and groups matching public target and option combinations.
5. Public answer events and private feedback events are emitted from the structured resolution.

## AW-283 handoff

AW-282 does not generate answers, decide truth, or update a knowledge graph. AW-283 receives InteractionResolution and composes answer content and contradiction metadata within the existing model-routing and knowledge-state boundaries.
