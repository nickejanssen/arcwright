# Requirements

## Platform Architecture Principles

These eight principles govern every technical decision. When a proposed requirement conflicts with one of these, the principle wins unless the founder explicitly overrides it with documented rationale.

**1. Surface agnosticism.** The engine has no opinion about what displays its output. It produces content events tagged with context: who the event is for, what triggered it, what the current session state is. The game layer built on top of the platform decides whether that event renders on a television, a phone, a web browser, a voice interface, or something that does not exist yet. Nightcap's TV-plus-phone setup is a Nightcap decision, not a platform decision. The platform never knows what a television is.

**2. Human arc primacy.** AI is the runtime personalization layer, not the creative author. Every experience running on Arcwright has a human-designed arc: the structural spine that defines what kind of experience this is, what the constraints are, what must happen and what can vary. The engine makes each session of that arc unrepeatable for the specific group inside it.

AI generation is called only after the arc execution layer has resolved the current session state deterministically. The LLM composes narrative from resolved, structured state data; it does not manage, infer, or update session state itself. State transitions are always deterministic. Any proposed feature that requires the AI to determine what has happened in a session, rather than express what the arc execution layer has already resolved programmatically, is a platform architecture violation and must be redesigned before implementation.

**3. Configurable composition.** Developers define each element of their arc as either authored (fixed) or generative (AI-driven at runtime). In Nightcap, the arc structure is fixed but the killer identity, character personalities, clue distribution, plot twists, and pacing are generative. In the monster RPG, the world rules are fixed but the player's story, the NPCs they encounter, and the world they explore are generative. This per-element composition dial is the core product differentiator.

**4. Unified character model.** Every character in an Arcwright session, whether controlled by a human player or driven by the AI, is the same type of object in the platform. Each character has an identity, a personality profile, goals, a knowledge state, and a relationship graph. What differs is the behavior source: AI characters have a behavior engine; human characters have an input channel.

**5. Knowledge graph as first-class infrastructure.** Every session maintains a knowledge graph: a structured record of who knows what, when they learned it, and from whom. AI character responses are automatically constrained by that character's knowledge state. This is always included in the platform, not an optional add-on. Schema complexity is what is tiered by plan.

**6. Cost-aware architecture from day one.** The platform routes AI model calls internally based on task type and quality tier. Classification and safety tasks route to small, fast, cheap models, including open source models served via managed inference providers (Groq, Fireworks, Together AI). Generation tasks route to capable models appropriate for the quality tier. Developers do not specify models; they specify task type and quality expectation. Chat 6 must calculate the per-session cost break-even between managed API pricing and open source self-hosting at projected session volumes before any self-hosting infrastructure is designed.

Two additional requirements apply from the first production session. First, routing is budget-first by default: runtime narrative generation defaults to the most cost-effective model meeting the quality tier for the task, not to the best available model. Frontier models are reserved for specific, documented, high-complexity arc junctions where the quality improvement is observable and justified. The routing default is cost-optimized-first; escalation to higher-cost models requires explicit justification in the routing configuration. Second, prompt caching is a named architectural requirement, not an assumed optimization: repeated system and context layers across a session (arc schema, character profiles, session rules, rolling session history) must be cached at every model provider that supports it. Cached input tokens cost a fraction of standard input tokens. Failing to cache repeated context in a session-based product wastes a predictable and avoidable portion of per-session AI cost from the first deployment. Chat 6 must specify which context layers are cached, at which providers, and how cache invalidation is handled when session state changes.

**7. Progressive proprietary infrastructure.** The platform launches on third-party managed AI services and replaces components with proprietary or open source self-hosted infrastructure as session volume, data quality, and economics justify each transition.

- *Tier 1, built at MVP:* Deterministic infrastructure with no AI cost: arc execution engine, session state management, knowledge graph, multi-surface event routing, developer API, usage tracking.
- *Tier 2, triggered by volume and cost pressure (Horizon 2 at earliest):* Specialized models fine-tuned on Arcwright session data. Requires thousands of structured, labeled sessions. Open source base models (Llama, Mistral, Phi) are the fine-tuning targets. Do not trigger before data and economics justify it.
- *Tier 3, never:* Foundation model development, compute infrastructure ownership.

Every completed session is a data asset. Session telemetry, instrumented from the first production session, is the mechanism that makes the Tier 2 transition possible. Competitors using the same APIs cannot replicate Arcwright-specific session data.

**8. Provider-agnostic model routing.** No platform operation has a hard dependency on any specific AI provider, model name, or model version. Every model call routes through an internal abstraction layer that maps task type and quality tier to the current best available option. This enables cost optimization through provider switching, compliance flexibility for enterprise customers, and the Tier 2 migration without changing any arc execution logic.

## Nightcap: Reference Implementation v1

Nightcap is an AI-powered murder mystery party game with a v1 floor of four human players, scaling up with additional human players. One player among the group is secretly the killer. v1 killer assignment is constrained-random behind the existing assignment interface. Two- and three-player sessions depend on interrogatable AI participants that can fill empty player slots, hold private knowledge state, and be the killer; that capability is deferred to v1.1. The arc structure is fixed for Nightcap; the content is generated fresh for each group.

**Platform capabilities Nightcap stress-tests:** Multi-character ensemble management across a structured arc with a fixed endpoint; per-player knowledge state enforcement; dynamic pacing; role assignment through a deterministic assignment interface; two-surface content routing via the Jackbox model.

**Arc skeleton (human-authored, fixed):** A murder has occurred before the session begins. The group is a set of characters with designed relationships and conflicting motives. Nightcap v1 uses the canonical eight Story Circle beats defined in `docs/story-bibles/nightcap-murder-mystery.md` Section 4: The Arrival, The Body, The Opening Move, The Dig, The Thread, The Reckoning, The Close, and The Truth. The reveal is a designed story moment. Beat count is an arc-level property; the platform and engine do not assume any fixed beat count.

**Generative layer (AI-driven, unique per session):** Killer identity and behavioral profile resolved through the assignment interface; character personalities and relationship tensions calibrated to the specific group; clue content, distribution sequence, and targeting; plot twists and misdirections based on group progress; narrator dialogue and atmosphere. Nightcap v1 includes group intake as personalization input: three host seed questions at session creation plus one to two player prompts at join. These inputs feed character tension calibration, generated character fit, and narrator callbacks.

**Mini-games:** Nightcap v1 includes timed, puzzle-gated mini-games across beats as a core interaction layer. Mini-games gate clue access, drive competition, and create investigative leads through the clue architecture. v1 captures behavioral-read output in schema but does not wire those signals into killer assignment; v1.1 wires mini-game behavioral signals into killer assignment, expands the mini-game suite, and adds cross-campaign behavioral reads.

**Nightcap Continuity:** Nightcap Continuity is named v1.1 scope, not v1. It includes cross-session group memory, narrator callbacks to prior sessions, and a shareable post-session recap. v1.1 continuity should also learn from a group's past sessions to deepen future intake personalization while preserving deterministic arc execution.

**Player experience requirements:** No app install required. Onboarding under two minutes. Each player's phone functions as their private game surface. A shared display carries the narrator and group-visible story moments. The host controls session pacing.

**Session persistence:** On interruption, the platform saves character state and knowledge state. On resume, the arc restarts from the nearest designed beat. No session restarts from the beginning.

**Success criteria:**

**Qualitative gate (required to consider v1 shipped):** Five or more completed sessions with real groups outside the founder's immediate circle where players finish the full arc and express unprompted enthusiasm. "It worked" is not the bar. "They wanted to play again" is the bar.

**Personalization perception gate (required alongside qualitative gate):** In at least two of the five qualifying sessions, at least one player says or does something unprompted that specifically references a personalized detail, not a generic arc beat. A player who says "that was fun" does not satisfy this gate. A player who reacts to a specific character tension, clue detail, or plot element that was clearly calibrated to their group does satisfy it. If the qualitative bar is met but the personalization perception gate is not, the platform's personalization layer is not working at the product level regardless of technical correctness. That result requires investigation and diagnosis before scope expands.

**Player count scope for first qualifying sessions:** Initial proof sessions target 4 to 6 human players. Nightcap v1 requires a four-human-player floor and scales up with additional human players. Two- and three-player sessions require interrogatable AI participants and are deferred to v1.1. The 4 to 6 human-player range reduces session coordination complexity, reduces per-session AI cost, and produces a cleaner diagnostic signal from early playtests before the system is hardened.

**Quantitative instruments (not the gate):** Session completion rate, arc pacing variance, killer concealment rate, replay intent.

## Future Reference Implementations

These are not scoped for MVP. Every significant platform architecture decision must be validated against them before it is committed. Each game is a deliberate platform stress-test, selected partly based on which capabilities it forces the platform to develop that prior games did not cover. Each game's design brief must include an explicit list of platform capabilities it is intended to validate. If a proposed game validates nothing new about the platform, it is a creative decision only and should be labeled as such.

**Monster RPG (Horizon 2, design TBD in Chat 8).** An emergent narrative experience with a designed world but no designed story. Platform capabilities this validates that Nightcap does not: world state persistence independent of any individual session; procedural generation at the world layer in addition to generative AI at the narrative layer; dynamic difficulty adjustment; player-defined motivation inference; simultaneous multi-player world state with no single designed endpoint.

**Enterprise Team-Building (Horizon 2).** A portfolio of enterprise arc templates running on the same Arcwright platform that powers Nightcap consumer. Arcs may be purpose-built for leadership development, team formation, onboarding narratives, or adapted from Nightcap structure with corporate-context content. The platform is shared; the products are distinct. Platform capabilities this validates: developer-configurable content rails strict enough for enterprise HR contexts without platform-level changes, multi-team session architecture, and debrief beat as a designed arc element type.

**Third game (selection deferred).** Should be selected deliberately against the platform capability gaps that Nightcap and the monster RPG do not cover. Candidates: asynchronous play, very short sessions under ten minutes, solo adaptive experiences, real-time competitive mechanics. Selection criterion is which gap in the platform's validated capability set is most important to close before opening to external developers.

## User and Developer Experience Requirements

**Developer experience.** At H1, a developer building on Arcwright receives a REST API with the knowledge graph as the headline primitive, a TypeScript web SDK auto-generated from OpenAPI, a hosted dashboard with the live knowledge graph inspection surface, documentation leading with knowledge graph value proposition, and Nightcap as the public reference implementation. The API accepts arc definitions as structured input and returns content events during session runtime. It exposes endpoints for session management, character management, knowledge state management, and content event streaming. The dashboard serves inspection, arc monitoring, and usage management before becoming a full visual editor. At H2 prep, Unity SDK, voice/audio SDK helpers, onboarding flow, and sample projects are staged. At H2 closed beta, Unreal SDK is added if demand signal exists, with multi-tenant features. At H3 plus, full Visual Storyworld phases, marketplace if pull exists, and AI-assisted authoring are pursued.

**Player experience (via Nightcap).** No account required. No app install required; phone browser is sufficient. Session join via code or QR under 30 seconds. Private information on the player's device must never appear on the shared display by design. Character identity and personality are delivered at session start.

**Host experience (via Nightcap).** Session creation and onboarding managed through the host interface on the shared display. Host can pause, advance manually, or allow the engine to drive pacing. Host has visibility into which players have received which clues without seeing clue content belonging to other characters. Host can remove a dropped player.

## What the Platform Must Do

**Session management:** Create, start, interrupt, resume, and end sessions. Emit content events tagged with target audience. Save state on interruption; restore from nearest designed beat on resume.

**Character management:** Maintain a unified character object for every participant. Generate AI character dialogue consistent with personality, goals, and knowledge state. Maintain ensemble coherence across a session arc.

**Knowledge graph:** Track per-character knowledge state throughout the session. Enforce that AI responses never reference information outside that character's knowledge state. Support configurable schema complexity.

**Arc execution:** Execute generative arc layers according to the developer-defined composition model. Adapt pacing based on session signals. Enforce authored constraints regardless of AI generation output.

**Safety:** Enforce platform-level hard stops at the model call layer, not the output filter layer. Serve developer-configurable content rails. Log safety filter activations. Make safety configuration visible in the dashboard.

**Cost and billing:** Track AI credit consumption per session, per arc, per developer account. Route model calls through the provider-agnostic abstraction layer. Enforce plan-tier limits and trigger overage or upgrade flows. Expose usage data for per-session gross margin calculation. Nightcap's temporary pricing north star is host-pays-per-session, anchored above the roughly $30 static murder-mystery-kit price point. The engine must support per-session charging and per-session cost tracking; the specific numeric launch price is deferred until real per-session AI cost and early willingness-to-pay data are available.

**Session telemetry:** Log structured telemetry for every session from the first production deployment. Minimum signals: arc beat engagement duration, pacing intervention triggers and outcomes, content event delivery confirmations, knowledge state constraint activations, session completion status, replay intent indicators.

**Narrative fidelity (ADR-0012):** Arc definitions may declare optional authored intent metadata: theme, tone, and per-beat emotional targets. When declared, the engine injects the block into generation context and logs realized-versus-intended pacing per beat. The engine tracks narrative obligations (authored setups and any pacing-injected misdirection) as durable session state and exposes a generic all-mandatory-obligations-resolved condition that arcs may use as a beat exit condition. Continuity is verified offline by an eval suite measuring knowledge-leak rate and character contradictions on synthetic sessions. The eval suite is pre-M6 hardening scope; intent and obligations implementation is sequenced post-M6 under epic M5-H.
